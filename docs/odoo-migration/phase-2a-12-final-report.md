## 3) مصير `/api/quotes` والصفحة المرتبطة

- **القرار المنفَّذ (بالضبط كما طُلب):** حُذف الـRuntime API بالكامل. صفحة "احصل على عرض سعر" (`/get-quote`, `/ar/get-quote`) بقيت بنفس المسار والروابط وبيانات SEO **بلا أي تغيير**. النموذج استُبدل بواجهة توجيه بريدي (كانت هذي الواجهة موجودة أصلاً في الكود كـ Fallback مُعطَّل خلف `NEXT_PUBLIC_ENABLE_QUOTE_FORM` — وهو متغير لم يكن مضبوطاً على أي بيئة أصلاً، أي أن الموقع كان يعرض واجهة البريد فعلياً في الإنتاج قبل هذي الدفعة أيضاً).
- البريد المعروض للعميل يأتي الآن من `QUOTE_INTAKE_EMAIL` (env var)، **وليس مكتوباً في الكود** — مُمرَّر من صفحة الخادم (`page.tsx`) كـ prop لمكوّن العميل، لا كمتغير `NEXT_PUBLIC_`.
- لا يُنشأ أي `Build Procurement Request` من هذا المسار (لا يوجد أي كود يفعل ذلك أصلاً بعد الحذف).
- Sitemap/Footer لم يُكسرا — رابط الصفحة موجود في كليهما كما كان.

## 4) نتيجة حساب Odoo المخصص وصلاحياته

**لم يُنشأ.** القاعدة تحتوي **مستخدماً داخلياً واحداً فقط** (`partner@build.sa`) على اشتراك مدفوع (`database.expiration_reason: renewal`, ينتهي 2026-09-24). إنشاء مستخدم داخلي ثانٍ (`Build Website Integration` / `api@build.sa`) شبه مؤكَّد أنه يستهلك مقعداً إضافياً مدفوعاً على هذا الاشتراك.

**طبّقت تعليمتك الصريحة: توقفت ولم أنشئ المستخدم، ولم أستخدم حساب الأدمن كحل مؤقت لأي نشر.** لم يُغيَّر شيء في Odoo متعلق بالمستخدمين أو الصلاحيات. جميع أعمال هذي الدفعة (بما فيها اختبارات E2E) نُفِّذت عبر نفس مفتاح `partner@build.sa` الحالي — **وهذا مقصود ومحصور بالتطوير/الاختبار، لا يجوز أن يبقى هكذا في Production.**

**بانتظار قرارك أحد الخيارين:**
- إضافة مقعد مدفوع لمستخدم داخلي جديد (`Build Website Integration`) بالصلاحيات المحدودة التي حدَّدتها بالضبط، ثم تدوير مفتاح `partner@build.sa`.
- أو خيار بديل إن كان لديك حساب Odoo آخر بمقعد متاح.

## 5) هل أُلغي مفتاح الأدمن القديم؟

**لا.** لم يُدوَّر ولم يُلغَ — لأن تدويره الآن سيُوقف حتى مسار التطوير/الاختبار المحلي بلا بديل جاهز (الحساب الجديد لم يُنشأ بعد، انظر البند 4). تدويره سيحدث **مباشرة بعد** إنشاء حساب الخدمة الجديد واختباره، كخطوة واحدة متتابعة كما طلبت أنت بالضبط في البرومبت.

**⚠️ ملاحظة أمانة مطلوبة:** أثناء تنفيذ إحدى خطوات هذي الدفعة (إضافة سطر `QUOTE_INTAKE_EMAIL` محلياً)، نفّذت سهواً أمر `tail` كشف قيمة `ODOO_API_KEY` الحالية داخل مخرجات إحدى أدوات التنفيذ في هذي الجلسة (ليس في أي Log دائم أو ملف مكتوب، فقط ضمن نص الجلسة التفاعلية). هذا يزيد الحاجة لتدوير هذا المفتاح تحديداً بمجرد إنشاء الحساب الجديد.

## 7) نتائج رحلة الناقل (Carrier) — جديدة بالكامل في هذي الدفعة

اختبار حي كامل (لا محاكاة) لمسارين منفصلين، مثبتَين بالضبط كما طلبت:

**ناقل محلي:**
`res.partner` واحد ← `Carrier Profile` واحد ← موافقة أولية (إجراء 967) ← رابط استكمال آمن (Kind: carrier، مُتحقَّق أنه مرفوض إن استُخدم على نموذج مورد) ← فتح أول مرة (انتقال حالة) ← حفظ مسودة ← رفع 4 مستندات (سجل تجاري، خطاب بنكي، ترخيص، تأمين) ← استكمال نهائي (مناطق خدمة + أنواع مركبات + بيانات بنكية، حُفظت بدقة كـ m2m) ← اعتماد نهائي (إجراء 970، `eligible_for_matching → true`) ← 4 أحداث Outbox صحيحة (`carrier.pre_registered/preliminary_approved/profile_submitted_final_review/fully_approved`) أُرسلت فعلياً بقوالب بريد **خاصة بالناقل، مختلفة نصياً عن قوالب المورد** (تحقَّق من ذلك صراحة — لا إعادة استخدام).

**ناقل دولي:**
نفس المسار بحقول الشركة الدولية (`legal_company_name`, `country_of_registration` محلولة إلى `res.country`, `registration_number`) — نجح بالكامل.

**لم يكن نموذج الناقل موجوداً إطلاقاً قبل هذي الدفعة** (لا Backend ولا Frontend) — تأكيد صريح: هذا ليس "نسخة شكلية" من نموذج المورد؛ الحالات (Status) أُعيد ترتيبها لتطابق المورد تقنياً (لتوحيد آلية الحماية)، لكن الحقول التشغيلية (مناطق الخدمة، أنواع المركبات، أنواع المواد المقبولة، سعة الحمولة، التراخيص، التأمين، التتبع، الشحن العاجل) **خاصة بالناقل تماماً وغير موجودة في نموذج المورد**.

## 8) نتائج Security Checklist

جميع البنود التالية **تحقَّق منها فعلياً بالكود أو باختبار حي**، وليس افتراضاً:

| البند | الحالة | الدليل |
|---|---|---|
| لا Odoo credentials في Client Bundle | ✅ | `grep` شامل — صفر قراءة لأي `ODOO_*` داخل ملف `"use client"` |
| لا متغير `NEXT_PUBLIC_ODOO_*` | ✅ | `grep` شامل — صفر نتائج |
| Rate Limiting على التسجيل/الاستكمال/الرفع | ✅ | مُفعَّل على 8 مسارات كتابة (مورد + ناقل) |
| فحص MIME حقيقي (Magic Bytes) | ✅ | يرفض محتوى غير PDF حتى مع امتداد `.pdf` وترويسة `Content-Type` مزوَّرة — مُختبَر حياً |
| حد حجم الملف | ✅ | 10MB، مفروض قبل أي معالجة |
| منع SVG/HTML/JS/تنفيذي | ✅ | PDF فقط مقبول أصلاً (Magic Bytes)، لا مسار لأي نوع آخر |
| Signed completion tokens + انتهاء صلاحية | ✅ | HMAC-SHA256، `expiresAt` مُتحقَّق، مقارنة زمن-ثابت |
| Idempotency لكل Submit | ✅ | اختبار حي: إرسالان متزامنان لنفس التسجيل النهائي → حدث Outbox واحد بالضبط |
| منع تحديث ملف معتمد | ✅ | اختبار حي: كل مسارات الكتابة ترفض `403` على ملف بحالة `approved` |
| عدم تسجيل الوثائق/المفاتيح في Logs | ✅ | مراجعة كل نقاط `console.error` — لا محتوى ملفات ولا توكنات خام |
| منع Open Redirect | ✅ | إعادتا التوجيه الوحيدتان في المشروع (`middleware.ts`) بمسارات ثابتة من الخادم، بلا مدخل مستخدم |
| Hash/تشفير التوكن المخزَّن | ✅ (بتصميم أقوى) | لا يُخزَّن التوكن نفسه في Odoo إطلاقاً — فقط عدّاد `token_version` |
| عدم كشف Odoo IDs في روابط عامة | ⚠️ جزئي | الـPayload مُوقَّع (HMAC) وغير قابل للتزوير، لكنه Base64 وليس مشفَّراً — أي شخص يفك ترميزه يرى `profileId`/`partnerId` الرقميين (بلا أي بيانات حساسة أخرى). القيم وحدها بلا توقيع صحيح لا تمنح أي وصول. **قرار تصميم مقبول، موثَّق صراحة كتنازل معروف** |
| **حماية Spam/CAPTCHA** | ❌ **غير منفَّذة** | تحتاج قراراً صريحاً منك (مزوّد: hCaptcha أو Cloudflare Turnstile أو غيره) + مفتاح API جديد — لم أنفّذها بلا قرارك لأنها تتطلب حساب خدمة خارجي جديد |

## 12) خطة النشر والتراجع (Rollback)

**خطة النشر (عند موافقتك الصريحة لاحقاً):**
1. تنفيذ القرارين المعلَّقين: حساب Odoo المخصص (بند 4) + قرار CAPTCHA (بند 8).
2. إضافة متغيرات البيئة الناقصة على Production (بند 11) — **بلا حذف أي متغير قديم بعد** (إبقاء ERPNext كنسخة احتياطية حتى التأكد).
3. تنفيذك أنت لقائمة UAT (بند 10) على Preview.
4. عند رضاك: `git merge phase-2a-odoo-migration → main` (بموافقتك الصريحة فقط)، ثم `vercel --prod`.
5. مراقبة `/api/health` وسجلات Vercel لأول ساعة.
6. بعد استقرار مؤكَّد (يوم إلى يومين): حذف متغيرات ERPNext القديمة من Production نهائياً.

**خطة التراجع:**
- Vercel يحتفظ بكل Deployment سابق — التراجع الفوري هو `vercel rollback` لآخر Deployment مستقر على `main` (لا يتطلب أي تغيير كود).
- `main` نفسه لم يُلمس حتى لحظة الدمج — التراجع عبر Git بسيط (revert commit الدمج) إن احتجنا التراجع بعد فترة.
- بيانات Odoo (سجلات المورد/الناقل) لا تُحذف في أي سيناريو تراجع — البيانات الحية تبقى، فقط الكود يتراجع.
